# Teacher-skill 项目学习笔记：PM 与工程师视角

> 更新日期：2026-05-02  
> 目的：从产品经理和工程师两个视角复盘当前项目，明确已经学到什么、项目价值在哪里、短板是什么、下一步该怎么迭代。

---

## 1. 项目一句话理解

Teacher-skill 是一个启发式数字助教 CLI。

它不是简单地“帮用户总结材料”，而是把学习过程拆成：

```text
输入学习材料 -> AI 分析切片 -> 逐个讲解 -> 提问验证 -> 判卷反馈 -> 保存进度
```

核心目标是解决“理解幻觉”：

用户看完解释后以为自己懂了，但其实无法复述、无法应用、很快遗忘。

这个项目的价值不在于“回答问题”，而在于“逼近真实理解”。

## 2. PM 视角：这个项目在解决什么问题

### 2.1 目标用户

当前 PRD 定义的目标用户是：

- 初级从业者
- 产品、运营、研发转型人群
- 需要自学 AI、技术文档、论文、复杂材料的人
- 容易被 AI 总结误导，以为自己已经理解的人

这类用户的共同问题不是“没有资料”，而是：

- 看得懂表面解释，但讲不出来
- 读完后缺少主动回忆
- 没有检验机制
- 不知道自己到底哪里没懂
- 学习过程缺少反馈闭环

### 2.2 核心产品假设

项目背后的产品假设是：

如果 AI 不只是给答案，而是先讲解、再提问、再根据回答反馈，用户会更容易形成真实理解。

这个假设可以拆成三层：

| 层级 | 假设 | 对应功能 |
|---|---|---|
| 学习材料层 | 用户有一段想学的材料 | 文本输入、文件输入 |
| 理解验证层 | 用户需要被提问才能发现盲区 | 讲解后提问、判卷 |
| 反馈修正层 | 用户答错后需要逐步提示 | hint_level 1-4、`/direct` |

### 2.3 产品亮点

当前项目最好的产品设计是“学习闭环”：

```text
讲解 -> 提问 -> 用户作答 -> 判卷 -> 反馈 -> 下一知识点
```

这比普通 ChatGPT 问答更有学习价值，因为它把用户从“被动看答案”拉回到“主动回忆和表达”。

第二个亮点是 `/direct`：

- 它允许用户在卡住时直接看答案
- 但会把知识点标记为 `needs_review`
- 这在产品上相当于承认用户有现实任务压力，同时保留学习记录

第三个亮点是进度持久化：

- 用户可以恢复历史 topic
- 每个 topic 有独立状态
- 学习不是一次性对话，而是有长期状态的流程

### 2.4 当前最大产品矛盾

当前产品定位和实际入口之间有矛盾：

目标用户偏非技术，但使用方式偏技术：

```text
pip install -> 配置 .env -> 填 API Key -> 命令行运行
```

所以当前版本更准确的定位应该是：

```text
面向开发者和技术学习者的 CLI 技术预览版
```

不应该急着包装成普通用户产品。

## 3. PM 视角：用户体验缺陷

### 3.1 入门门槛高

普通用户可能卡在：

- 不会安装 Python
- 不会使用命令行
- 不知道 API Key 是什么
- 不理解 `.env` 和 `config.yaml`

这说明项目工程上可用，但还没有产品化。

短期解决方案：

- README 清楚标明这是 CLI 技术预览版
- 推荐使用 `--file` 而不是手动粘贴
- 写清楚安装步骤和配置边界

长期解决方案：

- Web UI
- 拖拽上传文件
- 图形化配置
- 托管模型服务，隐藏 API Key

### 3.2 输入体验不自然

当前手动输入材料需要：

```text
粘贴文本
输入 /done 结束
```

这个交互对非技术用户不友好。用户会疑惑：

- 我粘贴成功了吗？
- 什么时候该输入 `/done`？
- 多行文本会不会丢？
- 中途想追加怎么办？

后续应把文件输入作为主路径：

```bash
python main.py --file article.md
```

并继续增强：

- `/load path`
- 中途追加材料
- 输入后显示材料摘要，让用户确认

### 3.3 用户控制感不足

现在流程按 chunk 顺序强制推进，用户不能自由控制：

- 不能跳过当前知识点
- 不能回到上一个知识点
- 不能查看全部知识点列表
- 不能跳到指定知识点

这会让用户感觉自己被流程推着走。

后续应优先增加：

```text
/skip      跳过当前知识点
/back      回到上一个知识点
/list      查看全部知识点
/jump 3    跳到第 3 个知识点
/review    查看待巩固知识点
```

### 3.4 反馈情绪不够柔和

答错时如果只是显示“错误”和提示，会让学习者有挫败感。

更好的反馈结构是：

```text
先接住用户：你抓到了一部分重点
再指出问题：但因果关系还没说清楚
再给提示：这是第 2 层提示，我给你一个类比
最后鼓励继续：你可以再试一次
```

提示等级本身是好的，但应该让用户知道：

- 当前是第几层提示
- 为什么给这个提示
- 后面还会怎么帮助他

### 3.5 进度感不足

当前 `/progress` 如果只显示 `3/7`，信息不够。

用户真正关心的是：

- 哪些已经掌握
- 哪些待巩固
- 哪些还没学
- 本轮答对多少
- 本轮答错多少
- 当前知识点标题是什么
- 还大概需要多久

后续 `/progress` 应从“位置提示”升级成“学习仪表盘”。

### 3.6 topic 名称不可读

如果历史主题叫：

```text
topic_20250502_143022
```

用户无法知道里面是什么内容。

应改成：

```text
2026-05-02 Transformer 入门
2026-05-02 PRD 写作学习
2026-05-02 推荐系统基础
```

这需要 topic 元数据支持：

- `title`
- `summary`
- `created_at`
- `last_active_at`
- `source_type`
- `source_path`

## 4. 工程师视角：当前系统结构

### 4.1 主流程入口

核心入口是：

```text
main.py
```

`TeacherSkillApp` 负责 CLI 编排：

- 检查 LLM 是否可用
- onboarding 摸底
- 选择历史 topic 或创建新 topic
- 加载文件或接收材料
- 调用 engine 分析材料
- 进入学习循环
- 保存进度
- 显示 summary

可以理解为：

```text
main.py = 用户交互层 + 应用编排层
```

它不应该承担太多教学逻辑。教学判断应该放在 `src/core/engine.py`。

### 4.2 教学状态机

核心教学逻辑在：

```text
src/core/engine.py
```

`TutorEngine` 负责：

- onboarding
- 材料分析
- 当前 chunk 讲解
- 接收答案
- 判卷
- 错误提示
- `/direct`
- 推进到下一个 chunk

状态由 `TutorState` 管理。

这是项目最重要的工程核心，因为它决定“学习闭环”是否稳定。

### 4.3 数据模型

主要模型在：

```text
models/protocol.py
models/state.py
models/user.py
```

核心概念包括：

| 模型 | 作用 |
|---|---|
| `AIMessage` | AI 返回给用户的统一消息格式 |
| `UserAnswer` | 用户回答结构 |
| `TopicState` | 一个学习主题的整体状态 |
| `ChunkState` | 单个知识点的状态 |
| `LearningStatus` | 知识点状态，如掌握、待巩固 |
| `UserProfile` | 用户画像和学习偏好 |

这些模型的价值是把 LLM 的不稳定输出转成可控的程序状态。

### 4.4 LLM 客户端与解析器

LLM 相关代码分两层：

```text
src/llm/client.py
src/llm/translator.py
```

`client.py` 负责和模型服务通信：

- API 调用
- 超时处理
- 鉴权错误
- 限流错误
- thinking 内容提取
- base_url 兼容

`translator.py` 负责把模型返回解析成结构化数据：

- JSON code block 提取
- 普通 JSON 解析
- 混合文本里的 JSON 提取
- 判卷结果降级判断
- 文本转 `AIMessage`

这个分层很重要：

```text
client 解决“怎么拿到模型输出”
translator 解决“怎么相信和使用模型输出”
```

### 4.5 存储与恢复

持久化在：

```text
src/utils/storage.py
data/users/
```

当前支持：

- 保存 topic 状态
- 保存 conversation history
- 保存 user profile
- 保存 learning index
- 列出历史 topics

这让项目不只是一次性脚本，而是一个有学习进度的应用。

### 4.6 配置系统

配置由两部分组成：

```text
.env
config.yaml
```

边界应该保持清晰：

| 文件 | 放什么 |
|---|---|
| `.env` | API Key、可选 base_url |
| `config.yaml` | model_id、temperature、max_tokens、prompt_mode、路径 |

这次已经校准过：

```text
模型和生成参数不要放在 .env
```

原因是 `.env` 更适合密钥，`config.yaml` 更适合可读、可版本管理的配置。

### 4.7 测试体系

当前单元测试覆盖：

- config
- storage
- file_loader
- memory
- rewards
- translator
- main flow

已有 102 个单元测试，本地和 GitHub Actions 已通过。

测试的意义不只是“证明代码没坏”，更重要的是保护核心学习流程：

- 恢复 topic 不应重复启动教学
- 最后一题答对后应退出学习循环
- `/direct` 后也应正确推进或结束
- Windows pytest 不应因为输出流替换崩溃

这些都是发布阻塞级问题，已经修复。

## 5. PM 与工程的共同判断

### 5.1 当前项目已经达到什么程度

当前可以认为达到：

```text
v0.2.0 工程可用版
```

具备：

- 可安装
- 可配置
- 可运行
- 可测试
- 有 CI
- 有文档结构
- 有标准 Skill 结构
- 核心教学闭环可用

但还没有达到：

```text
普通用户产品可用版
```

因为：

- 安装门槛高
- CLI 输入体验弱
- 学习控制命令不足
- 进度展示不够直观
- topic 历史不可读
- 反馈情绪设计不足

### 5.2 下一阶段最重要的产品主题

下一阶段不应该优先做 Persona。

更重要的是：

```text
让用户获得控制感
```

也就是先解决：

- 我能不能跳过
- 我能不能返回
- 我能不能查看列表
- 我能不能修改答案
- 我能不能知道自己学到哪
- 我能不能看懂历史主题

Persona 是体验增强，控制感是基础体验。

### 5.3 下一阶段最重要的工程主题

工程上应该把 `main.py` 里的 CLI 控制逻辑逐步模块化。

可以考虑拆出：

```text
src/cli/commands.py
src/cli/renderer.py
src/core/navigation.py
```

原因：

- `/skip`、`/back`、`/jump` 会让主循环变复杂
- 进度展示会变成独立的渲染逻辑
- topic 元数据会影响 storage 和 main
- 如果都堆在 `main.py`，后面会难维护

## 6. 建议发布计划

### v0.2.0：工程可用版

目标：

```text
GitHub 上可安装、可运行、可测试
```

应包含：

- 核心教学闭环
- 文件输入
- 进度保存
- README 入口
- 安装文档
- Release checklist
- GitHub Actions 单测通过
- 标准 Skill 结构

发布文案要明确：

```text
当前是 CLI 技术预览版，不是普通用户 GUI 产品。
```

### v0.3.0：CLI 可用性修复版

目标：

```text
让用户在 CLI 中有控制感
```

优先做：

- `/skip`
- `/back`
- `/list`
- `/jump`
- `/review`
- 更好的 `/progress`
- topic 可读标题
- 答案提交确认
- 错误反馈优化

### v0.4.0：复习模式

目标：

```text
从一次性学习变成长期学习
```

优先做：

- 记录薄弱知识点
- 复习待巩固内容
- 跳过讲解直接提问
- 根据历史错误排序
- 复习后更新学习状态

### v0.5.0：教学风格 Persona

目标：

```text
让不同用户获得不同风格的教学体验
```

可以做：

- 严师模式
- 温和导师模式
- 苏格拉底模式
- 同伴辅导模式

### v1.0.0：普通用户产品版

目标：

```text
降低非技术用户门槛
```

需要考虑：

- Web UI
- 文件拖拽上传
- 可视化进度
- 历史学习列表
- 一键继续学习
- 不让用户直接处理 API Key 和命令行

## 7. 从这个项目可以学到的能力

### 7.1 PM 能力

通过这个项目可以学习：

- 如何定义目标用户
- 如何从痛点推导产品机制
- 如何区分“工程可用”和“用户可用”
- 如何判断功能优先级
- 如何识别产品定位和使用门槛的矛盾
- 如何写 PRD、路线图、发布清单、状态文档

最重要的一点：

```text
好功能不等于好体验。
```

当前核心闭环是好功能，但用户控制感不足，所以还需要体验迭代。

### 7.2 工程能力

通过这个项目可以学习：

- Python CLI 项目结构
- Pydantic 数据建模
- LLM API 调用封装
- LLM 输出解析与降级策略
- 状态机设计
- 文件存储与恢复
- 配置管理
- 日志系统
- 单元测试
- GitHub Actions CI
- 文档工程化

最重要的一点：

```text
LLM 应用的核心工程难点不是“调用模型”，而是“把不稳定的模型输出变成稳定的软件行为”。
```

### 7.3 Agent Skill 能力

这个项目也可以学习如何把经验沉淀成 Skill：

- `skills/heuristic-teacher/SKILL.md` 只放稳定工作流
- `docs/` 放 PRD、路线图、学习笔记
- `references/` 放补充说明
- 不把个人学习笔记污染到 Skill 核心

这个边界很重要：

```text
Skill 是给 Agent 执行的稳定说明。
docs 是给人理解项目的上下文。
learning-notes 是给自己学习和复盘的笔记。
```

## 9. 一句话总结

Teacher-skill 当前最值得学习的地方，是它把一个抽象的学习痛点“理解幻觉”，落成了一个可运行、可测试、可持续迭代的 LLM 教学系统。

它现在不是完美产品，但已经是一个很好的学习样本：

```text
PM 学产品判断，工程师学 LLM 应用架构，个人开发者学如何把想法推进到可发布状态。
```
