## PM 视角：这个项目现在解决了什么问题

### 1. 核心用户问题

这个项目最开始要解决的问题是：

> 用户看完 AI 的解释，以为自己懂了，但真正要复述、应用、答题时发现其实没掌握。

这叫“理解幻觉”。

普通 AI 问答通常是：

```text
用户问问题 → AI 直接解释 → 用户感觉懂了 → 结束
```

Teacher-skill 的设计是：

```text
用户输入材料 → AI 拆知识点 → 逐个讲解 → 提问验证 → 判卷反馈 → 再继续
```

这个变化很重要。

因为学习不是“看解释”，而是“经过验证”。  
只有用户能回答问题，才说明他真的开始理解。

### 2. 当前产品定位

现在的 Teacher-skill 更适合定位为：

> 面向愿意使用命令行的学习者、开发者、技术学习者的本地启发式学习工具。

它不是普通大众产品。

因为目前还需要：

- 安装 Python
- 配置 `.env`
- 填 API Key
- 用命令行运行

所以从 PM 视角要清楚：

```text
当前版本适合开发者/学习者试用
不适合直接面向完全不懂技术的普通用户发布
```

这不是失败，而是阶段定位。

先把核心学习闭环跑通，再考虑降低门槛，这是合理路线。

### 3. 当前完成的产品能力

#### 核心学习闭环

已经完成：

```text
讲解 → 提问 → 用户回答 → 判卷 → 正确进入下一题 / 错误给提示
```

这是整个产品最核心的价值。

没有这个闭环，项目只是一个“文档总结器”。  
有了这个闭环，它才是一个“教学工具”。

#### 文件学习入口

已经支持：

- `.md`
- `.txt`
- `.pdf`

并且推荐使用：

```bash
python main.py --file article.md
```

这比让用户手动粘贴一大段材料更合理。

从 PM 视角看，这一步解决的是“用户如何开始”的问题。

#### 学习控制感

已经支持：

- `/progress` 查看进度
- `/list` 查看全部知识点
- `/review` 查看待巩固知识点
- `/skip` 跳过当前知识点
- `/back` 回到上一个知识点
- `/jump N` 跳到指定知识点
- `/direct` 直接看答案
- `/load <路径>` 中途追加材料
- `/history` 查看历史学习记录

这些命令解决的是“用户有没有控制感”的问题。

如果用户只能被系统强制按顺序走，他会觉得被困住。  
现在用户可以跳过、返回、查看、追加材料，体验已经接近一个真正工具。

#### 答案提交确认

现在用户输入答案后不会立刻判卷，而是先确认：

- Enter 提交
- `/edit` 修改
- `/cancel` 取消

这个功能看起来很小，但它非常重要。

因为 CLI 里很容易出现：

- 粘贴错内容
- 多打了字
- 回答没写完
- 换行被截断

如果直接判卷，用户会觉得系统“不讲理”。  
确认机制给了用户一个后悔机会。

#### 错误反馈情绪优化

之前答错可能像这样：

```text
❌ 你的答案不对
```

现在改成更温和的反馈面板，并显示提示层级：

```text
还差一点，我们换个角度试试
提示层级：第 2/4 层（生活类比）
```

这背后的 PM 思考是：

> 学习产品不能只判断对错，还要保护用户继续学习的意愿。

对学习工具来说，用户连续答错很正常。  
如果系统太硬，用户会放弃。  
所以反馈要像陪练，而不是像考试扣分。

#### 历史学习记录

现在已经新增：

- `LearnedTopic`
- `UserProfile.history_topics`
- 完成主题后自动归档
- `/history` 查看历史记录

这一步是复习模式的地基。

因为如果系统不知道用户以前学过什么，就没法支持：

```text
复习一下之前学过的 Transformer
```

所以 `/history` 不是最终功能，而是为后面的“智能复习”准备数据。

## 工程视角：现在的系统结构是什么样

### 1. 当前核心目录

项目结构大致是：

```text
Teacher-skill/
├── main.py                         # CLI 入口，处理用户交互
├── models/                         # 数据模型
│   ├── protocol.py                 # AIMessage / ResponseType
│   ├── state.py                    # TopicState / ChunkState
│   └── user.py                     # UserProfile / LearnedTopic
├── src/
│   ├── core/
│   │   ├── engine.py               # 教学状态机
│   │   ├── memory.py               # 对话记忆
│   │   └── rewards.py              # 连击激励
│   ├── llm/
│   │   ├── client.py               # LLM 调用
│   │   ├── translator.py           # JSON / AIMessage 解析
│   │   └── exceptions.py           # LLM 错误分类
│   └── utils/
│       ├── config.py               # 配置加载
│       ├── file_loader.py          # 文件读取
│       ├── logger.py               # 日志
│       └── storage.py              # JSON 存储
├── prompts/                        # 分层 Prompt
├── skills/heuristic-teacher/       # 标准 Skill 定义
├── tests/                          # 单元测试和集成测试
└── docs/                           # 文档与学习笔记
```

### 2. main.py 的职责

`main.py` 主要负责 CLI 交互。

它做的是：

- 启动程序
- 检查 API Key
- 跑 onboarding
- 选择新主题或恢复旧主题
- 加载文件
- 接收用户命令
- 显示表格、面板、进度
- 保存学习进度
- 学完后归档历史记录

可以理解为：

```text
main.py = 用户界面层 / CLI 控制层
```

它不应该承担太多“教学判断逻辑”。  
真正的教学流程在 `TutorEngine` 里。

### 3. engine.py 的职责

`src/core/engine.py` 是核心状态机。

它负责：

- 分析材料
- 构建知识点
- 讲解当前知识点
- 接收答案
- 调用 judge prompt 判卷
- 更新 chunk 状态
- 进入下一知识点
- 支持 skip/back/jump

可以理解为：

```text
engine.py = 学习流程发动机
```

它不直接关心界面怎么显示，而是关心：

```text
当前是什么状态？
用户做了什么？
下一步应该去哪？
```

### 4. 状态模型的重要性

现在核心状态模型是：

```text
TopicState
  └── chunks: List[ChunkState]
```

一个主题里面有多个知识点。

每个知识点有：

- 标题
- 内容
- 问题
- 正确答案
- 当前状态
- 答错次数
- 提示层级
- 尝试次数

这让系统可以知道：

```text
哪些学会了
哪些没学会
哪些需要复习
用户卡在哪一层提示
```

这也是后续复习模式的基础。

### 5. UserProfile 的演进

之前 `UserProfile` 主要记录：

- 用户水平
- 熟悉的话题
- 总主题数
- 完成主题数

现在新增了：

```python
history_topics: List[LearnedTopic]
```

每个 `LearnedTopic` 记录：

- topic_id
- 标题
- 摘要
- 完成时间
- 总知识点数
- 掌握数量
- 待巩固数量
- 来源文件

这说明项目的数据结构正在从“当前学习状态”扩展到“长期学习档案”。

从工程角度看，这是一个很关键的升级。

因为后续要做复习、推荐、薄弱点分析，都需要长期学习档案。

### 6. Prompt 分层架构

项目现在已经有分层 Prompt：

```text
_base.md
_persona.md
system/
  00_onboarding.md
  01_analyzer.md
  02_teach.md
  03_judge.md
```

这样做的好处是：

- 通用规则放 `_base.md`
- 教学人格放 `_persona.md`
- 不同任务有独立 Prompt
- 后续可以单独加复习 Prompt

这比把所有规则写在一个大 Prompt 里更可维护。

### 7. 测试和 CI

项目已经有：

- 单元测试目录
- 集成测试目录
- GitHub Actions workflow

这说明工程上已经从“能跑就行”进入到“改完要验证”的阶段。

当前要注意的是：

```text
最近 CLI 可用性和复习地基改动后，需要重新跑 pytest tests/unit/
```

因为只要改了主流程、状态模型、用户档案，就必须确认：

- 旧功能没坏
- 新功能有测试覆盖
- CI 能通过

## 已推进与还没完成的任务

### 1. 复习意图识别

已完成基础版：

```text
用户输入：复习一下 transformer
系统识别：这是复习意图
```

这是 v0.4 的入口。

当前采用关键词匹配：

```python
REVIEW_KEYWORDS = ["复习", "回顾", "温习", "再看一遍"]
```

先用关键词匹配，不急着上 LLM，这是合理的低成本方案。

### 2. 主题匹配

已完成基础版：

```text
用户输入 transformer
系统从 history_topics 中找到最接近的 topic
```

当前已经用简单规则完成：

- 匹配 title
- 匹配 summary
- 匹配 source_path 文件名

不需要一开始就做复杂语义匹配；后续如果主题很多，再考虑向量检索或 LLM 语义匹配。

### 3. 复习引擎

已完成基础版：

- 跳过讲解
- 直接提问
- 优先复习 `needs_review`
- 根据复习答题结果更新状态

这是 v0.4 最核心的工程任务。当前已在 `TutorEngine.start_review()` / `next_review_chunk()` 中实现复习队列，薄弱点优先。

### 4. 复习 Prompt

已完成：

```text
prompts/system/04_review.md
```

它告诉模型：

- 这是复习，不是首次学习
- 不要长篇讲解
- 直接提问
- 答错给简短提示
- 优先薄弱点

### 5. 普通用户产品化

还没完成：

- Web UI
- 拖拽上传
- 可视化进度
- API Key 配置引导
- 普通用户无需命令行

这是更后面的 v1.0 方向。

## 给自己的学习总结

这个项目教会我的几个关键点：

### 1. 产品不是只做功能，而是做闭环

“讲解”只是功能。  
“讲解 → 提问 → 判卷 → 提示 → 复习”才是闭环。

闭环越完整，产品价值越强。

### 2. 用户体验问题要具体拆

“不好用”不能直接修。

要拆成：

- 输入门槛高
- 无法修改答案
- 不能跳过
- 不能回退
- 看不到进度
- 答错反馈太硬

拆细以后，每个问题都可以对应一个工程任务。

### 3. 工程结构要服务后续扩展

现在把文档、Skill、Prompt、模型、引擎分开，是为了以后能继续扩展。

如果所有东西都堆在 `main.py` 或 `SKILL.md` 里，后面会越来越难维护。

### 4. 先做可验证的小步

这次没有一口气做完整复习模式，而是先做：

```text
history_topics + /history
```

这是正确节奏。

因为大功能要拆成能测试、能提交、能解释的小功能。
