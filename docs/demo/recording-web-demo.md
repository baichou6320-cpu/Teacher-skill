# Teacher-skill 网页录屏演示流程

这份文档用于把项目磨成一个稳定、可重复、适合给领导看的短视频 Demo。目标不是展示所有功能，而是在 60-90 秒内让对方看懂三件事：

- 你能把一个 AI 学习想法做成可以运行的产品原型。
- 这个原型不是普通聊天，而是有资料、卡片、问答、判别、复习状态的学习闭环。
- 你能同时处理产品体验、前端交互、后端 API、测试和演示稳定性。

## 启动方式

在项目内层目录执行：

```powershell
cd C:\Users\Administrator\Desktop\Teacher-skill\Teacher-skill
python main.py --web --recording-demo
```

启动后会自动打开：

```text
http://127.0.0.1:8765/?demo=1
```

如果浏览器已经打开旧页面，请强制刷新一次。这个模式会加载固定的演示材料和固定知识卡片，适合录屏时稳定复现。

录屏演示模式走 `demo_server.py`，只依赖 Python 标准库。它不需要 API Key，不需要真实 LLM 响应，也不需要先把完整依赖装好。正式学习模式仍然需要 `requirements.txt` 里的依赖和 API 配置。

## 录屏前检查

```powershell
python main.py --check
python -m pytest tests/unit/ -q
```

录屏时不要展示 `.env`、API Key、终端历史里的密钥，也不要临时修改配置。网页演示模式优先使用固定离线数据，避免现场网络或模型响应不稳定。

## 推荐录制顺序

1. 打开网页，展示三栏布局：左侧资料，中间学习对话，右侧学习状态。
2. 点击“开始演示学习”，让系统生成固定知识卡片。
3. 展示右侧知识卡片和学习进度，说明材料已经被拆成可验证的学习单元。
4. 在中间输入框发送示例问题：`不太能，可以直接跟我说这个概念。`
5. 展示系统给出的两种路径：渐进引导和直接解释。
6. 选择其中一种路径，说明项目会根据用户意图切换学习方式。
7. 最后扫一眼 GitHub Actions 绿灯或本地测试结果，证明项目不是只靠界面截图。

## 60-90 秒口播稿

开头：

“这个项目叫 Teacher-skill，它解决的是学习里的一个真实问题：很多人以为自己看懂了资料，但其实没有形成可验证的理解。所以我做的不是普通总结工具，而是把学习材料变成一个可验证的学习闭环。”

资料和卡片：

“左边是资料区，用户可以粘贴文本或上传资料。系统会把资料拆成知识卡片，每张卡片都有标题、验证问题、参考答案和学习状态。这样用户不是只看一段 AI 总结，而是一步步确认自己是否真的理解。”

对话学习：

“中间是学习对话区。用户可以回答问题，也可以直接说自己不懂。系统会判断用户现在是想被引导，还是想先听详细解释，然后切换对应的教学方式。”

学习状态：

“右边是学习状态区，会显示当前阶段、知识卡片、掌握情况和待巩固内容。这个设计的重点是让学习过程可追踪，而不是聊完就结束。”

工程收尾：

“工程上我做了 CLI 入口、本地网页入口、后端 API、固定录屏 Demo、单元测试和 GitHub Actions。也就是说它已经从一个想法，变成了可以运行、可以验证、可以继续迭代的产品原型。”

## 录屏时不要展示的内容

- 不展示 `.env` 文件。
- 不展示真实 API Key。
- 不现场安装依赖。
- 不现场处理 Git 冲突。
- 不录太长的命令行报错过程。

## 翻车备用方案

如果端口被占用：

```powershell
python main.py --web --web-port 8766 --recording-demo
```

然后打开：

```text
http://127.0.0.1:8766/?demo=1
```

如果网页看起来还是旧版本：

```powershell
Invoke-RestMethod http://127.0.0.1:8765/api/health
```

确认返回内容里包含：

```text
demo
recording_demo
```

如果没有，说明浏览器连接的是旧后端进程，需要关闭旧进程后重新启动。

## 录屏评价重点

领导不一定关心每一行代码，但会关心这些信号：

- 你能发现真实用户痛点：看懂不等于会用。
- 你能把痛点拆成产品流程：资料输入、知识拆解、问答验证、复习闭环。
- 你能把流程落地成工程：CLI、Web、API、测试、CI。
- 你能为展示场景做稳定性设计：固定演示模式、固定数据、可重置流程。
